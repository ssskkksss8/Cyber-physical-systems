## 1. Выбор начальных условий

### 1a) Набор данных
Используется **Massachusetts Roads Dataset** - датасет для задачи **семантической сегментации дорог** на аэрофотоснимках.

### Почему выбран именно он
Цель в автоматическом выделении дорожной сетки на спутниковых и аэрофотоснимках.
Это много где применяется. Например в:
- картографировании;
- мониторинге инфраструктуры;
- транспортной аналитике;
- планировании логистики;
- навигации автономных систем;
- анализе последствий чрезвычайных ситуаций.

### Что требуется от модели
Для каждого пикселя изображения модель должна определить, относится он к классу **road** или к **background**.

---
## 1b) Метрики качества

Для оценки качества выбраны:

- **Dice coefficient** — основная метрика, особенно важная для тонких структур по типу дорог;
- **IoU (Jaccard Index)** — стандартная метрика качества сегментации;
- **Pixel Accuracy** — дополнительная метрика общей точности.

### Обоснование выбора

Ключевыми метриками являются **Dice** и **IoU**, так как именно они лучше отражают качество перекрытия предсказанной маски дороги с эталонной разметкой.

## 2. Создание бейзлайна и оценка качества

В качестве бейзлайна использовались модели из `segmentation_models.pytorch` со следующими условиями:

- resize изображений;
- отсутствие аугментаций;
- функция потерь `BCEWithLogitsLoss`;
- optimizer `AdamW`;
- обучение в Google Colab на GPU.

### Бейзлайн-модели
В качестве библиотечных моделей использовались:

- **U-Net + ResNet34 encoder**
- **FPN + ResNet34 encoder**

### Полученные результаты baseline

#### 1. U-Net + ResNet34
Метрики достаточно устойчивые.

Итоговые метрики:
- **Dice ≈ 0.4728**
- **IoU ≈ 0.3158**

По кривым обучения видно, что:
- loss снижался;
- Dice и IoU стабильно росли;
- модель начала выделять дорожную сеть достаточно корректно.

#### 2. FPN + ResNet34
В данном эксперименте архитектура показала слабый результат.

Итоговые метрики были близки к нулю:
- **Dice ≈ 0.0007**
- **IoU ≈ 0.0004**

### Вывод по baseline
Лучшей baseline-моделью оказалась **U-Net + ResNet34**, тогда как **FPN + ResNet34** в выбранной конфигурации показала неудовлетворительное качество.

Результаты baseline сохраняются в:
- `artifacts/baseline_results.csv`

## 3. Улучшение бейзлайна

### 3a) Сформулированные гипотезы

Были выдвинуты следующие гипотезы:

1. Аугментации (отражения, повороты, сдвиги, изменение яркости) улучшат обобщение модели.
2. Комбинация `BCE + Dice Loss` сработает лучше, чем один `BCE`, потому что дороги — тонкие и разреженные объекты.
3. Планировщик `ReduceLROnPlateau` сделает обучение стабильнее.
4. Предобученные энкодеры дадут результат сильнее, чем обучение с нуля.

### 3b) Проверка гипотез
В improved pipeline были добавлены:

- аугментации;
- `BCE + Dice Loss`;
- scheduler;
- те же архитектуры для честного сравнения с baseline.

### 3c) Результаты improved baseline

#### 1. Improved U-Net + ResNet34
Улучшенный пайплайн дал небольшой, но реальный прирост качества по сравнению с baseline U-Net.

Итоговые метрики:
- **Dice ≈ 0.4897**
- **IoU ≈ 0.3253**

Это означает, что улучшенный pipeline действительно помог повысить качество сегментации.

#### 2. Improved FPN + ResNet34
Для FPN улучшенный pipeline не дал хорошего результата. Архитектура осталась существенно слабее U-Net.

### Вывод по improved baseline
Улучшенный pipeline оказался **полезным для U-Net**, но **не помог FPN**.  
Следовательно, эффективность выбранных техник зависит не только от задачи, но и от архитектуры модели.

Результаты improved baseline сохраняются в:
- `artifacts/improved_results.csv`

## 4. Имплементация алгоритма машинного обучения

В соответствии с требованием задания, в ноутбуке реализована **собственная сегментационная модель** без использования `segmentation_models.pytorch`.

### Реализованная модель
**TinyUNet** — компактная encoder-decoder архитектура, реализованная вручную на `PyTorch`.

### Архитектура модели
Модель включает:

- блоки `Conv2d + BatchNorm + ReLU`;
- downsampling через `MaxPool2d`;
- bottleneck;
- upsampling через `ConvTranspose2d`;
- skip-connections;
- финальный `1x1 Conv` для предсказания бинарной маски.

### Выполненные эксперименты
В ноутбуке TinyUNet обучалась в двух режимах:

1. **TinyUNet baseline**
2. **TinyUNet improved**

### Полученные результаты

#### 1. TinyUNet baseline
Собственная baseline-модель показала **лучший результат среди всех экспериментов**.

Итоговые метрики:
- **Dice ≈ 0.6267**
- **IoU ≈ 0.4607**

#### 2. TinyUNet improved
После применения улучшенного pipeline качество оказалось ниже, чем у baseline-версии TinyUNet.

Итоговые метрики:
- **Dice ≈ 0.5296**
- **IoU ≈ 0.3613**

### Вывод по собственной реализации
Собственная модель **TinyUNet baseline** оказалась сильнее библиотечных baseline-моделей и сильнее improved U-Net.  
При этом улучшенный pipeline не дал прироста для собственной архитектуры, а наоборот ухудшил её результат.

Результаты сохраняются в:
- `artifacts/manual_results.csv`
